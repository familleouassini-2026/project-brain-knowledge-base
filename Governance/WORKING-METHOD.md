# Project Brain Working Method

**Document ID**: GOV-001  
**Version**: 1.0.0  
**Status**: Draft  
**Owner**: Project Brain  
**Last Update**: 2026-07-03

## Purpose

This document defines the working method used to build Project Brain.

## Core rule

Conversation is for reasoning.

Release is for building.

## Release cycle

Each release must contain:

- a version number;
- a changelog;
- release notes;
- updated roadmap;
- concrete files;
- clear next step.

## Session rule

Each working session must aim to produce one of the following:

- a document;
- a decision;
- a release;
- a prototype;
- a tested feature;
- a correction.

## Anti-drift rule

No new methodology or architecture concept should be added unless it meets three criteria:

1. It solves a current problem.
2. It simplifies the project.
3. It can be implemented quickly.

Otherwise it goes to the backlog.

## Product Owner rule

The Product Owner should not be asked to perform repetitive technical tasks when these tasks can be packaged into a release.

## Architect rule

The architect must protect coherence and delivery.

This includes stopping over-analysis when a useful release can already be produced.
